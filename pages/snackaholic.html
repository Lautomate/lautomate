Tools and Techniques: Excel, Python, and SQL

Topic: Snack Transactions

This project aims to reduce data entry errors and enhance business operations. While some data entry practices may seem outdated, I am here to provide a modern solution that respects traditional operations. Please note that this project replicates the actual business scenario but does not use the company's real data.

My first step was to gather all the requirements and identify the columns needed for the project. I created a simple SQL table for products, inserting all 31 products along with their interconnected column values. Next, I developed a table for shop areas, filling in missing information to provide more context, such as city, province, region, and island, since the available data only included barangay-level information. Subsequently, I created a comprehensive transactions table that encompasses all the necessary columns, identical to those in the Excel that I will build later on.

After establishing all the required tables and inputting predefined values, I designed a standardized format in Excel. This format aims to minimize data entry while maintaining data integrity and accuracy. I maximized the use of data validation to streamline their workflow.

I utilized a few but powerful Python libraries for data handling, database interaction, and packaging. Data entry specialists will now only need to input minimal data into Excel, run the standalone Python application, and have the data securely stored in the database.

I understand that when data is pushed from Excel, the transactions table may contain incomplete information. This is because I broke down the data into three distinct tables to simplify data entry in Excel. The first step in my data transformation process involves modifying the InvoiceNo column values based on the payment method. Since SQL does not interpret the custom formats from Excel but rather the actual input, this step is crucial.

Finally, I performed SQL joins on the transactions table to include all records from the Transactions table along with the matching records from the Products and Areas tables, which I then fed into views.

The rationale behind breaking the data into different tables and eventually joining them is to cater to various stakeholders who engage in different processes. My goal is to provide high-granularity data while reducing their effort and increasing productivity.
